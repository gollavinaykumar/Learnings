# 📖 Why Do We Need Signals?

Imagine a process is running.

```
Python Program

↓

Running
```

Suddenly:

- User presses **Ctrl + C**
- System is shutting down
- Parent process exits
- Child process finishes
- Timer expires

How can Linux notify the process?

The answer is:

```
Signal
```

---

# What is a Signal?

A Signal is a small asynchronous notification sent to a process.

```
Kernel

↓

Signal

↓

Process
```

or

```
Process A

↓

Signal

↓

Process B
```

Signals notify,

they usually **do not carry data**.

---

# Signal Flow

```
Sender

↓

Kernel

↓

Target Process

↓

Signal Handler

↓

Action
```

The kernel is responsible for delivering the signal.

Now press:

```

```


Flow:

```
Keyboard

↓
```

# SIGINT

Meaning:

```
Interrupt
```

Usually generated by:

```
Ctrl + C
```

Default action:

```
Terminate
```

Applications may catch it.

---

# SIGTERM

Meaning:

```
Terminate Gracefully
```

Default action:

```
Terminate
```

Unlike `SIGKILL`,

applications can intercept it.

Typical behavior:

```
Receive SIGTERM

↓

Save Data

↓

Close Files

↓

Exit
```

---

# Why SIGTERM Is Preferred

Suppose PostgreSQL receives:

```
SIGTERM
```

It can:

✔ Finish transactions

✔ Flush data

✔ Close connections

✔ Exit safely

---

# SIGKILL

Meaning:

```
Kill Immediately
```

Default action:

```
Immediate Termination
```

The kernel immediately destroys the process.

No cleanup.

No handlers.

---

# Why Can't SIGKILL Be Ignored?

Suppose malware ignored every kill request.

The operating system could never stop it.

Therefore:

```
SIGKILL

↓

Always Works
```

Applications cannot block,

ignore,

or handle it.

---

# SIGSTOP

Stops a process immediately.

Example:

```bash
kill -STOP 1234
```

Process state:

```
Stopped
```

The process remains in memory,

but no CPU time is allocated.

---

# SIGCONT

Continues a stopped process.

Example:

```bash
kill -CONT 1234
```

Flow:

```
Stopped

↓

SIGCONT

↓

Running
```

---

# SIGHUP

Originally:

```
Terminal Disconnected
```

Modern usage:

```
Reload Configuration
```

Example:

```
Nginx

↓

SIGHUP

↓

Reload nginx.conf

↓

Continue Serving
```

No restart required.

---

# SIGCHLD

Suppose:

```
Parent

↓

Child
```

Child exits.

The kernel sends:

```
SIGCHLD
```

to the parent.

The parent can now call:

```
wait()
```

to collect the child's exit status.

Suppose:


# Pending Signals

Signals aren't always delivered immediately.

```
Signal Sent

↓

Pending

↓

Process Scheduled

↓

Signal Delivered
```

The kernel stores pending signals until the process can receive them.

---

# Signal Masks

A process can temporarily block certain signals.

Example:

```
Critical Section

↓

Block SIGINT

↓

Finish Work

↓

Unblock SIGINT
```

Useful for protecting sensitive operations.

---

# Real-Time Signals

Linux also supports:

```
SIGRTMIN

↓

...

↓

SIGRTMAX
```

Unlike traditional signals,

Real-Time Signals:

✔ Preserve order

✔ Can queue multiple instances

✔ Can carry additional information

# Docker Example

Suppose:


Docker does **not** immediately kill the container.

Instead:

```

↓

↓

↓
```

This allows applications to shut down gracefully.

# systemd Example

Exactly the same philosophy.

# Interview Questions

## What is a Signal?

A Signal is an asynchronous notification delivered to a process by the kernel or another process.

---

## What is the difference between SIGTERM and SIGKILL?

`SIGTERM` allows graceful shutdown and can be handled by the application.

`SIGKILL` immediately terminates the process and cannot be caught, blocked, or ignored.

---

## What signal is generated by Ctrl + C?

`SIGINT`.

---

## Why is SIGKILL uncatchable?

To ensure the operating system can always terminate a process when necessary.

---

## What is SIGHUP commonly used for today?

Reloading configuration files without restarting an application.

---

## Why does Docker send SIGTERM before SIGKILL?

To give the application an opportunity to shut down cleanly before forcing termination.

---

## What is SIGCHLD?

A signal sent to a parent process when one of its child processes exits or changes state.

---

# Summary

Signals allow Linux to notify and control running processes.

```
Sender

↓

Kernel

↓

Signal

↓

Target Process

↓

Signal Handler

↓

Action
```

Key concepts:

- Signals are asynchronous notifications.
- The kernel delivers signals to processes.
- `SIGINT` is generated by Ctrl + C.
- `SIGTERM` requests graceful shutdown.
- `SIGKILL` forces immediate termination.
- `SIGSTOP` pauses a process.
- `SIGCONT` resumes a stopped process.
- `SIGHUP` commonly reloads configuration.
- Docker and `systemd` rely heavily on signals for lifecycle management.

Understanding signals is essential before studying process scheduling, debugging, container runtimes, and advanced Linux kernel behavior.
